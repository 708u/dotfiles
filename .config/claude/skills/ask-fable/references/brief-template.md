# Fable brief テンプレート

`fable-brief.md` はこの構成で書く。全セクションを埋める。
埋める内容が無いセクションは「なし」と明記する（省略しない。
省略すると Fable がその観点を見落とす）。

```markdown
# Fable への brief

## タスク / ゴール
<今セッションで達成しようとしていること>

## 核心の問い
<Fable に答えてほしいことを 1〜2 文で>

## 議論の経緯
- 検討した選択肢:
- 決定事項:
- 未決事項:

## 調査で分かったこと
<結論と、その根拠（ファイルパス・コマンド出力）>

## 実装内容
<変更対象と要点。関連する変更のみ。diff を貼る場合も
今扱っている対象に絞る>

## 制約・前提
<Fable が知らないと誤評価する前提。リポジトリ固有のルール、
既存の設計方針、外部要因など。無ければ「なし」>

## 参照ファイル一覧
<Fable が Read すべき絶対パス。1 行 1 パス>
```

## 制約・前提に書かないこと

ユーザーの global CLAUDE.md は subagent に自動注入される。
その内容を「制約・前提」へ再掲しない。二重化になり、global 側の
変更で brief だけ古くなる。ここにはリポジトリ固有・セッション
固有の前提だけを書く。

## 自己完結チェック

brief を書き終えたら、次を確認してから subagent を起動する。

- 「先ほど」「上記」「例のあれ」のような、このセッションを
  見ないと解決しない指示語が残っていないか
- 核心の問いを読むだけで、何を評価してほしいか伝わるか
- 参照ファイル一覧だけで実装の実物にたどり着けるか
