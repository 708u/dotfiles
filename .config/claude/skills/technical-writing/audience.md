# 読者を意識する

## 基本方程式

```txt
良いドキュメント = 読者が必要とする知識 - 読者の現在の知識
```

過剰な情報は読者を圧倒し、不足した情報は理解を妨げる。

## 読者定義の3要素

### 1. 役割の特定

基本的な知識レベルが共有される職種を識別:

- ソフトウェアエンジニア
- 技術系非エンジニア（プロダクトマネージャー、テクニカルライターなど）
- 研究者
- 学生
- 非技術系ステークホルダー

### 2. 知識の近接性

考慮すべき要素:

| 要素 | 質問 |
|------|------|
| 専門分野への習熟度 | 読者はこの技術に詳しいか |
| 関連プロジェクト経験 | 類似システムを使ったことがあるか |
| 時間経過による知識減衰 | 最後に触れたのはいつか |

```txt
例:
チームのソフトウェアエンジニア → 深い知識を仮定可能
外部の開発者 → 基本的な説明が必要
非技術系マネージャー → 技術用語の解説が必要
```

### 3. 学習目標の明確化

読者が習得すべき具体的なタスクまたは情報をリストアップ:

```txt
このドキュメントを読んだ後、読者は以下ができるようになる:
- APIエンドポイントを設定する
- 認証トークンを取得する
- 基本的なCRUD操作を実行する
```

## 知識の呪い

### 問題

専門家は無意識に初心者向けに不適切な説明をしがち:

- 略語を説明なしに使用
- 実装詳細を前提知識として扱う
- 「明らか」「簡単」という表現を多用
- 重要なステップを省略

### 対策

| 症状 | 対策 |
|------|------|
| 略語の乱用 | 初出時に必ず定義 |
| 暗黙の前提 | 前提条件セクションを設ける |
| 抽象的な説明 | 具体例を追加 |
| 省略されたステップ | 初心者にテストしてもらう |

### 読者の視点に立つ方法

1. 文書を声に出して読む
2. 同僚（特に異なる専門分野）にレビューを依頼
3. 用語集を作成し、すべての専門用語を定義
4. 新入社員や初心者にテストしてもらう

## 国際対応のベストプラクティス

### 文化的中立性

避けるべき表現:

| 避ける | 理由 |
|--------|------|
| スポーツの比喩 | 文化によって異なるスポーツが人気 |
| 祝日への言及 | 国によって異なる |
| 政治的・宗教的参照 | 敏感なトピック |
| ユーモアやジョーク | 翻訳で失われる |

### ローカル参照の排除

```txt
悪い例: "a sticky wicket" (英国のクリケット用語)
良い例: "a challenging problem"

悪い例: "hit a home run" (野球用語)
良い例: "achieved a major success"
```

### イディオム（慣用句）の排除

慣用句は直訳すると意味が通じない:

| イディオム | 明確な表現 |
|-----------|-----------|
| out of the box | by default, without configuration |
| under the hood | internally, in the implementation |
| piece of cake | simple, easy |
| ballpark figure | approximate number |

### 単純で明確な言語

複雑な文構造を避ける:

```txt
悪い例:
The aforementioned functionality, which was implemented
in the previous release, notwithstanding certain edge cases,
performs optimally under normal operating conditions.

良い例:
This feature works well in most cases. Some edge cases
may not work as expected.
```

### 翻訳を考慮した執筆

- 短い文を使用
- 一貫した用語を維持
- 代名詞の参照を明確に
- 文化固有の参照を避ける

これらの原則は翻訳の品質を向上させるだけでなく、
英語が第一言語でない読者の理解も助ける。
