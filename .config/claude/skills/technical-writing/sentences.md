# 文章の書き方

## 能動態 vs 受動態

### 基本構造

| 態 | 構造 | 例 |
|----|------|-----|
| 能動態 | 主語 + 動詞 + 目的語 | The cat sat on the mat. |
| 受動態 | 目的語 + be動詞 + 過去分詞 | The mat was sat on by the cat. |

### 受動態の認識方法

受動態の動詞は通常「be動詞 + 過去分詞」で構成:

- was interpreted
- is generated by
- was formed by

### 能動態を優先する理由

1. 読者の処理時間を短縮
2. より直接的で明確な意思疎通
3. 行為者の曖昧性を回避
4. 一般的により簡潔

### 変換例

| 受動態 | 能動態 |
|--------|--------|
| A wrapper is generated by the registration process. | The registration process generates a wrapper. |
| The data was processed by the server. | The server processed the data. |
| Errors are logged by the system. | The system logs errors. |

### 受動態が適切な場合

- 行為者が不明または重要でない場合
- 行為者を意図的に省略したい場合
- 科学論文の慣習に従う場合

```txt
受動態が適切: The samples were collected over three months.
→ 誰が集めたかより、サンプル自体が重要
```

### 命令形は能動態

命令形で始まる文は主語(you)が省略された能動態:

```txt
Click the Submit button. → (You) click the Submit button.
Configure the settings. → (You) configure the settings.
```

## 明確な文章

### 強い動詞を選ぶ

弱い動詞（is, are, occur, happen）を具体的な動詞に置き換える。

| 弱い表現 | 強い表現 |
|---------|---------|
| The exception occurs when dividing by zero. | Dividing by zero raises the exception. |
| This error message happens when... | The system generates this error message when... |
| We are very careful to ensure... | We carefully ensure... |

強い動詞を選ぶことで、文章に具体性と明確さが生まれる。

### "There is/There are"の排除

これらの構文は汎用名詞と汎用動詞の組み合わせ:

| 悪い例 | 良い例 |
|--------|--------|
| There is a variable called X. | The X variable stores... |
| There are two facts you should know. | You should know two facts. |
| There is no guarantee... | Clients might not receive... |

### 曖昧な形容詞・副詞の排除

主観的な表現を客観的なデータに置き換える:

| 曖昧 | 具体的 |
|------|--------|
| screamingly fast | 225-250% faster |
| very large | 10GB |
| extremely reliable | 99.99% uptime |

これにより文書がマーケティング資料のようにならず、信頼性を維持できる。

## 短い文章

### 1文1アイデアの原則

各文は単一の考えに焦点を当てる。

```txt
悪い例:
The late 1950s was a key era for programming languages because
IBM introduced Fortran in 1957 and John McCarthy introduced
Lisp the following year, which gave programmers both an
imperative and a functional approach.

良い例:
The late 1950s was a key era for programming languages.
IBM introduced Fortran in 1957.
John McCarthy introduced Lisp the following year.
Consequently, programmers had access to both an imperative
and a functional approach.
```

### 長い文をリストに変換

"or"や複数の項目を含む文は箇条書きリストに変換:

```txt
変換前:
To configure the system, set the timeout, specify the retry
count, enable logging, and define the output directory.

変換後:
To configure the system:
- Set the timeout
- Specify the retry count
- Enable logging
- Define the output directory
```

### 不要な言葉の削減

| 冗長な表現 | 簡潔な表現 |
|-----------|-----------|
| causes the triggering of | triggers |
| at this point in time | now |
| in order to | to |
| for the purpose of | to, for |
| in the event that | if |

### that vs. which

| 用法 | 説明 | 例 |
|------|------|-----|
| which | 非本質的な従属節（コンマ前） | Python, which was invented by Guido, is popular. |
| that | 必須の従属節（コンマなし） | Calculations that don't involve algebra are simpler. |

"which"で始まる節を削除しても文の意味が通じる場合、それは非本質的。
"that"で始まる節を削除すると文の意味が変わる場合、それは必須。
