# イラストレーション

視覚的要素を効果的に活用するためのガイドライン。

## 基本原則

### キャプション優先アプローチ

イラストを作成する前にキャプションを書く。

優れたキャプションの特性:

- 簡潔
- 重要なポイントを説明
- 読者の注意を集中させる

```txt
悪い例: システム図
良い例: リクエストがロードバランサーを経由してサーバーに到達する流れ
```

### 情報量の制限

複雑なシステムは単一図式では表現しない。

ガイドライン:

- 1段落分の情報以上を1つの図に入れない
- 複雑なシステムはサブシステムに分割
- 段階的に詳細を説明

```txt
悪い例: 全アーキテクチャを1枚の図に
良い例:
  1. 概要図（高レベル）
  2. 各コンポーネントの詳細図
  3. データフローの図
```

### 視点の焦点化

読者の注意を関連部分に導くための視覚的手がかり:

| 手法 | 用途 |
|------|------|
| 赤い楕円 | 重要な部分をハイライト |
| コールアウト矢印 | 特定要素への注意喚起 |
| 番号付きラベル | 順序や参照を示す |
| 色分け | カテゴリや状態を区別 |

### スクリーンショットでの焦点化

スクリーンショットを使用する際:

1. 関連部分のみをキャプチャ
2. 重要な部分を視覚的にハイライト
3. 不要な情報はトリミングまたはぼかし

## 反復改善

ロンドン地下鉄路線図の進化が示すように、
イラストレーションは反復的修正を必要とする。

焦点を当てるべき点:

- 簡潔性
- 明確性
- アクセシビリティ

### 改善プロセス

```txt
1. 初期バージョンを作成
2. 意図が伝わるか確認
3. 不要な要素を削除
4. 視覚的階層を調整
5. フィードバックを収集
6. 改善を反映
```

## 図の種類と用途

| 種類 | 用途 | 例 |
|------|------|-----|
| フローチャート | プロセスや判断の流れ | ワークフロー、アルゴリズム |
| シーケンス図 | 時系列の相互作用 | API呼び出し、メッセージング |
| アーキテクチャ図 | システム構成 | コンポーネント関係 |
| スクリーンショット | UI操作手順 | 設定画面、操作例 |
| 表 | 比較、一覧 | 機能比較、設定オプション |

## ツール

### 推奨ツール

| ツール | 特徴 |
|--------|------|
| Google Drawings | 無料、共同編集可能 |
| diagrams.net | 無料、多機能 |
| LucidChart | 豊富なテンプレート |
| Mermaid | コードベースの図 |

### 出力形式

- **SVG**: 品質維持に最適、拡大縮小で劣化しない
- **PNG**: 広い互換性、固定解像度
- **PDF**: 印刷用途

## アクセシビリティ

### 代替テキスト

すべての図に意味のある代替テキストを提供:

```txt
悪い例: alt="図1"
良い例: alt="ユーザーリクエストがAPIゲートウェイを経由して
        バックエンドサービスに到達するシーケンス図"
```

### 色に依存しない

色覚多様性に配慮:

- 色だけで情報を伝えない
- パターンや形状も併用
- 十分なコントラストを確保

### テキストのサイズ

図中のテキストは十分な大きさで:

- 最小12pt推奨
- 拡大しても読める解像度
