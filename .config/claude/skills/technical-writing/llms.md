# LLMを技術文書作成に活用する

LLMは下書き生成、文書修正、フォーマット、要約に活用できる。
ただし、常に回答を注意深く確認する必要がある。

## 活用方法

### 1. 初期草稿の生成

プロンプトに含めるべき要素:

| 要素 | 説明 | 例 |
|------|------|-----|
| 役割設定 | 特定の立場を指定 | 「専門家の技術ライターとして」 |
| 対象読者 | 知識レベルを明確に | 「Python初心者向け」 |
| 文書タイプ | 形式を指定 | 「FAQセクション」「APIリファレンス」 |
| 目標設定 | 読者が習得すべきこと | 「認証フローを理解する」 |
| スタイル指定 | フォーマットの詳細 | 「箇条書きと短い段落を使用」 |

セクション単位で生成することを推奨。

### 2. 文書修正

LLMが得意とする修正:

- 文法、句読点、綴りの誤り検出
- 組織構造の改善案提示
- 受動態から能動態への変換
- スタイルガイド適合性のチェック
- 論理的矛盾や誤りの指摘

```txt
プロンプト例:
「以下のテキストをGoogle Technical Writingスタイルガイドに
従って修正してください。能動態を使用し、各文は1つのアイデアに
限定してください。」
```

### 3. フォーマット変換

異なるフォーマット間の変換が可能:

- Word文書 -> Markdown
- プレーンテキスト -> 構造化HTML
- リスト -> 表
- 長文 -> 箇条書き

### 4. 要約作成

LLMは長い文章を圧縮するタスクに適している。

プロンプトで指定すべき点:

- 目標の長さ（例: 50-100語）
- スタイル（フォーマル/カジュアル）
- 対象読者
- 目的（概要/エグゼクティブサマリーなど）

## 効果的なプロンプト作成

### 背景情報の提供

LLMの理解を向上させるため、以下を添付:

- ソースコード
- 既存のドキュメント
- トランスクリプト
- 図表の説明

### ハルシネーション対策

```txt
「提供したテキストからのみ情報を使用してください。
情報が不足している場合は、その旨を明記してください。」
```

### 反復改善

初回プロンプトは完全でなく、調整を重ねる必要がある:

```txt
1回目: 基本的な内容を生成
2回目: 「もっと具体的な例を追加して」
3回目: 「初心者向けに用語を説明して」
```

## 検証の必要性

### すべての回答を確認

LLMの出力は必ず検証する:

| 確認項目 | 方法 |
|---------|------|
| 事実の正確性 | 公式ドキュメントと照合 |
| コードの動作 | 実際に実行して確認 |
| リンクの有効性 | アクセスして確認 |
| 用語の一貫性 | 既存ドキュメントと比較 |

### 違和感チェック

- 不自然な表現がないか
- 文脈に合わない情報がないか
- 過度に一般的または曖昧な記述がないか

## 注意事項

### 執筆プロセスの価値

執筆は思考を明確にするプロセス。

LLMによる自動生成は以下の利益を損なう可能性がある:

- 深い理解の形成
- 論理的思考の訓練
- 独自の視点の発見

### 推奨アプローチ

```txt
1. まず自分でアウトラインを作成
2. 各セクションの要点を整理
3. LLMを補助として活用（表現の改善、例の追加）
4. 最終的な判断は人間が行う
```

### 機密情報の取り扱い

- 機密データをLLMに送信しない
- 社内情報は匿名化または架空データに置換
- 組織のAI利用ポリシーに従う

## ユースケース別ガイド

### README作成

```txt
「以下のコードのREADMEを作成してください:
- インストール方法
- 基本的な使い方
- 設定オプション
- 対象: このライブラリを初めて使う開発者」
```

### APIドキュメント

```txt
「以下の関数のAPIドキュメントを作成してください:
- パラメータの説明
- 戻り値
- 使用例
- エラーケース」
```

### 変更履歴

```txt
「以下のコミットログから変更履歴を作成してください:
- ユーザー向けの変更のみ
- 破壊的変更は明示
- バージョン番号ごとにグループ化」
```
